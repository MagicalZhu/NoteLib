服务网关-GateWay

1.GateWay​

1.1 GateWay 概述​

  • GateWay 是什么？

    • SpringCloud Gateway 使用的Webflux中的Reactor-Netty响应式编程组件，底层使用了Netty通讯框架
  • GateWay 可以干什么？
    • 反向代理
    • 鉴权
    • 流量控制
    • 熔断
    • 日志监控
    • ...
  • Gateway 特性
    • 动态路由: 可以匹配任何请求属性，可以对路由指定Predicate(断言) 和 Filter(过滤器)
    • 集成Hystrix断路器功能
    • 集成SpringCloud 服务发现功能
    • 易于编写Predicate 和 Filter
    • 请求限流功能
    • 支持路径重写

  • GateWay 和 Zuul1.x的区别

    • Zuul 1.x 是基于阻塞式IO 的API Gateway，基于 Servlet 2.5 使用阻塞式架构，不支持任何长连接(如websocket)
    • SpringCloud GateWay建立在 Spring5、Reactor和SpringBoot2.x 之上，使用非阻塞API

1.2 GateWay 三个核心概念​

  • 路由
    • 路由是构建网关的基本模块
    • 由ID、目标URI、一系列的断言和过滤器组成，如果断言为true则匹配该路由
  • Predicate
    • 路由转发的判断条件
    • 开发人员可以匹配HTTP请求中的所有内容（例如请求头或请求参数），如果请求与断言相匹配则进行路由
      • 目前SpringCloud Gateway支持多种方式，常见如：Path、Query、Method、Header等。
  • Filter
    • Spring框架中GatewayFilter的实例
      • 分为 pre[前置] 和 post [后置] 过滤器
    • 使用过滤器，可以在请求被路由前或者之后对请求进行修改

1.3 GateWay 工作流程​

  • 工作流程
    1. 客户端向Gateway 发送请求，然后在Gateway Handler Mapping 中找到与请求匹配的路由，然后将请求转发到 Gateway Web Handler
    2. Handler 再通过指定的过滤器链将请求发送到我们的实际业务层处理，然后返回
      • 在 前置过滤器(pre Filter)中，可以做 参数校验、权限校验、流量监控、日志输出、协议转化等
      • 在后置过滤器(post filter)中，可以做 响应内容、响应头修改，日志的输出、流量监控等
  • Gateway 自动注入的Bean

2.GateWay 使用​

2.1 路由​

  • 常见的类

    • RouteDefinition 路由定义信息,包含下面的几个属性

      • PredicateDefinition
        • 断言定义信息
      • FilterDefinition
        • 过滤器定义信息

    • RouteDefinitionLocator

      • 加载路由定义信息，常见有以下的实现类

        • CachingRouteDefinitionLocator

          • 基于HashMap缓存的路由定义信息加载器

        • DiscoveryClientRouteDefinitionLocator

          • 基于服务注册中心的路由定义信息加载器，通过ReactiveDiscoveryClient.getServices() 进而获取到注册中心的所有服务实例

          • ReactiveDiscoveryClient 常见的实现类有
            • ConsulReactiveDiscoveryClient
            • EurekaReactiveDiscoveryClient

          • 在 GatewayDiscoveryClientAutoConfiguration 中注册

        • InMemoryRouteDefinitionRepository

          • 基于内存的路由定义信息加载器

        • PropertiesRouteDefinitionLocator

          • 基于配置文件的路由定义信息加载器，比如application.yml

      • 这些路由定义信息加载器 会在 GatewayAutoConfiguration 中进行注册

  • 操作步骤

    1. 引入GateWay 和 服务注册中心的依赖 (GateWay也需要注册到注册中心哟)

      • 注意：使用GateWay的时候不要引入 spring-web 的依赖，因为GateWay 构建在 WebFlux  上

    2. 配置application.yml，通过spring.cloud.gateway.routes配置路由映射信息,是一个 List<RouteDefinition>

      • id
        • 路由ID,没有固定的规则但要求唯一，建议配合服务名
      • uri
        • 路由匹配后,提供服务的URI
      • predicates
        • 断言，判断请求的路由是否与之匹配
          • 是一个 Map<String,String>
        • 当客户端的请求路由与Predicates 匹配时，会将请求转发到 配置的>URI+请求PATH 上去

    3. 做了路由映射之后，既可以通过网关的uri 访问，也可以通过 路由地址的uri 访问

①、 引入依赖，不需要引入 web的依赖

②、 配置application.yml

server:
  port: 9527
spring:
  application:
    name: cloud-gateway-gateway-9527
  cloud:
    consul:
      host: localhost
      port: 8500

        service-name: cloud-gateway
        instance-id: ${spring.application.name}

      routes:
        - id: payment8003_route1      # 路由ID,没有固定的规则但要求唯一，建议配合服务名

          predicates:
            - Path=/payment/get/**   # 断言，路径相匹配的进行路由

2.2 配置路由​

  • GateWay提供两种方式配置路由
    1. 配置application.yml
    2. 向容器中注入 RouterLocator 的Bean

2.2.1 注入Bean 的方式配置路由​

  • GateWay 支持采用fluent api 的方式注入 RouterLocator Bean的方式向注入路由

  • 常见的类

    • RouteLocatorBuilder : 用于构建一个 RouteLocator 对象

      • builder.routes().route(String id, Function<PredicateSpec, Route.AsyncBuilder> fn)
        • id : 路由的id
        • fn: 函数式接口，用于构建一个fluent api 的路由
  • 操作步骤
    • 向容器中注入 RouterLocator

配置类中注入 RouterLocator 的Bean

@Configuration
public class RouteConfig {
    @Bean
    public RouteLocator routes(RouteLocatorBuilder builder){
        RouteLocatorBuilder.Builder routes = builder.routes();
        return routes.route("path_route_news",
                r->r.path("/*")
                     .uri("http://news.baidu.com")).build();
    }
}

2.2.2 通过微服务名实现动态路由​

  • 默认情况下Gateway会根据注册中心的服务列表，以注册中心上微服务名为路径创建动态路由进行转发，从而实现动态路由的功能

  • 操作步骤
    1. 配置application.yml
      • spring.cloud.gateway.discovery.locator.enabled
        • 是否开启从注册中心动态创建路由的功能，利用服务名进行路由
      • spring.cloud.gateway.routes.uri
        • 如果用服务名做路由，URL采用lb://serviceName的格式
      • spring.cloud.loadbalancer.ribbon.enabled
        • 是否启用Ribbon作为负载均衡,默认采用的是 SpringCloud LoadBalancer,且只有轮询负载

    ①、 配置application.yml

    server:
      port: 9527
    spring:
      application:
        name: cloud-consul-gateway-9527
      cloud:
        gateway:
          discovery:
            locator:
              enabled: true                             # 开启从注册中心动态创建路由的功能，利用服务名进行路由
          routes:
            - id: route_by_consul
              uri: lb://cloud-payment-service-consul    # 指定从那个服务中获取api服务
              predicates:
                - Path=/payment/**
        loadbalancer:
          ribbon:
            enabled: true                           # 采用Ribbon作为服务的负载均衡组件
        consul:
          host: localhost
          port: 8500
          discovery:
            service-name: cloud-consul-gateway
            instance-id: ${spring.application.name}

②、 启用Ribbon作为负载均衡组件后，注入随机负载均衡规则

@SpringBootApplication
public class GatewayConsulApplication {
    public static void main(String[] args){
        SpringApplication.run(GatewayConsulApplication.class,args);
    }

    @Bean
    public IRule rule(){
        return new RandomRule();
    }
}

    • 这些Predicate 会匹配HTTP请求的不同属性
    • 不同的Predicate可以组合，使用逻辑and组合

2.4 过滤器Filter​

  • 使用过滤器，可以在请求被路由前或者之后对请求进行修改
  • 过滤器分类
    • 按生命周期
      • pre : 前置过滤器，在执行业务逻辑之前调用
      • post: 后置过滤器，在执行业务逻辑之后调用
    • 按种类
      • Global Filter : 全局过滤器
      • Gateway Filter: 局部过滤器

  • 常见的Global Filter

2.4.1 自定义全局过滤器​

  • 有何作用？
    • 全局日志记录
    • 统一网关鉴权
    • ...
  • 常见的接口
    • GlobalFilter
      • 全局过滤器接口
    • ServerWebExchange
      • Web服务交换机
      • 提供对HTTP请求和响应的访问，还公开其他与服务器端处理相关的属性和特性，如请求属性
    • GatewayFilterChain
      • 全局过滤器链
  • 操作步骤
    • 创建类实现 GlobalFilter

自定义过滤器

/**
 * <b>请求拦截</b>
 * @author create by zyl
 * create at 2020/12/12 9:46 上午
 */

@Component
@Order(0)
@Slf4j
public class LogGatewayFilter implements GlobalFilter {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        log.warn("--------- call LogGatewayFilter --------------");
        String hostString = exchange.getRequest().getRemoteAddress().getHostName();
        String method = exchange.getRequest().getMethodValue();
        String URI = exchange.getRequest().getURI().getPath();
        log.warn("URI:{},method:{},hostString:{}",URI,method,hostString);
        return chain.filter(exchange);
    }
}
