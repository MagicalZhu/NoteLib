服务调用-Ribbon

1.Ribbon​

1.1 Ribbon 概述​

  • 什么是Ribbon ?
    • SpringCloud Ribbon是基于Netflix Ribbon实现的一套`客户端 负载均衡`的工具。
      • Ribbon是Netflix发布的开源项目，主要功能是提供客户端软件负载均衡算法，将Netflix的中间层服务连接在一起
    • Ribbon客户端组件提供一系列完善的配置项如连接超时，重试等
      • 即在配置文件中列出Load Balancer（简称LB）后面所有的机器，Ribbon会自动的帮助你基于某种规则去连接这些机器
        • 比如 : 简单轮询，随机连接等

      • 我们也很容易使用Ribbon实现自定义的负载均衡算法

1.2 Ribbon负载均衡​

  • 负载均衡(Load Balance) ?

    • 将用户的请求平摊的分配到多个服务上，从而达到系统的高可用性(HA)

      • 常见的负载均衡有软件 Nginx，LVS，硬件 F5等
    • 集中式 LoadBalance

    • 在服务的消费方和提供方之间使用独立的 LoadBalance 设施 ,由该设施负责把访问请求通过某种策略转发至服务的提供方

      • 可以是硬件，如F5
      • 也可以是软件，如nginx
    • 进程内 LoadBalance
      • 将 LoadBalance 逻辑集成到消费方, 消费方从服务注册中心获知有哪些地址可用，然后再从这些地址中选择合适的服务器
        • Ribbon属于进程内LB
        • Ribbon 只是一个类库，集成于消费方进程，消费方通过它来获取到服务提供方的地址

1.3 Ribbon 的组成​

  • 下面所有的方式都可以实现 细粒度的自定义配置 , 即向容器中注入自定义的 Bean
接口作用
IClientConfig读取配置
IRule负载均衡规则,选择实例
IPing筛选掉ping不同的实例
ServerList<Server>交给Ribbon的实例列表
ServerListFilter<Server>过滤掉不符合规则的实例
ILoadBalancerRibbon的入口
ServerListUpdater更新交给Ribbon的List 的策略

2.Ribbon 使用​

2.1 Ribbon 基本使用​

  • 操作步骤
    1. 引入SpringCloud Ribbon的依赖
      • 注: 在Eureka、Consul 中依赖了Ribbon组件，可以不引入Ribbon
    2. 通过配置类注入 RestTemplate 时，使用 @LoadBalanced注解标注
      • 该注解会将 RestTemplate 配置为一个具有负载均衡功能的 Http客户端
    3. 使用RestTemplate 发送Http 请求时,使用 服务的 ServiceName替换端口和ip

①、 通过@Loadbalanced 注解注入一个具有负载均衡的RestTemplate

  • 默认的负载均衡算法实现是 RoundRobinRule 轮询算法
@Configuration
public class OrderConfig {
    @Bean
    @LoadBalanced
    public RestTemplate restTemplate(){
        RestTemplate restTemplate = new RestTemplate();
        List<HttpMessageConverter<?>> list = restTemplate.getMessageConverters();
        // 替换 StringHttpMessageConverter 的默认字符集
        for (HttpMessageConverter<?> httpMessageConverter : list) {
            if(httpMessageConverter instanceof StringHttpMessageConverter) {
                ((StringHttpMessageConverter) httpMessageConverter).setDefaultCharset(StandardCharsets.UTF_8);
                break;
            }
        }
        return restTemplate;
    }
}

②、 使用RestTemplate

  • RestTemplate 此时具有负载均衡功能
@Slf4j
@RestController
@RequestMapping("/consumer")
public class OrderController {
    @Resource
    private RestTemplate restTemplate;
        
    // 使用服务的serviceName 替换掉原来的 ip：端口 
    private static final String CLOUD_PAYMENT_SERVICE_CONSUL="http://cloud-payment-service-consul";

    @GetMapping("/getPayments/consul")
    public String getPayments(){
        String result = restTemplate.getForObject(CLOUD_PAYMENT_SERVICE_CONSUL+"/payment/get", String.class);
        return result;
    }
}

2.2 Ribbon 核心组件IRule​

  • 负载均衡策略 根据特定算法中从服务列表中选取一个要访问的服务
  • Ribbon 内置的一些负载均衡策略
    • RoundRobinRule
      • 轮询策略 (默认使用的)
      • 请求接口的次数 % 服务器集群总数量 = 实际调用服务器位置下标
    • RandomRule
      • 随机策略
    • AvailabilityFilteringRule
      • 会先过滤掉由于多次访问故障而处于断路器跳闸状态的服务，还有并发的连接数量超过阈值的服务
        • 然后对剩余的服务列表按照轮询策略进行访问
    • WeightedResponseTimeRule
      • 根据平均响应时间计算所有服务的权重，响应时间越快服务权重越大被选中的概率越高。
      • 刚启动时如果统计信息不足，则使用RoundRobinRule策略，等统计信息足够，会切换到  WeightedResponseTimeRule
    • RetryRule
      • 先按照RoundRobinRule的策略获取服务，如果获取服务失败则在指定时间内会进行重试，获取可用的服务
    • BestAvailableRule
      • 会先过滤掉由于多次访问故障而处于断路器跳闸状态的服务，然后选择一个并发量最小的服务
    • ZoneAvoidanceRule
      • 复合判断Server所在区域的性能 和 Server的可用性选择服务器

2.2.1 全局注入Ribbon的负载均衡策略 (所有客户端共享)​

  • 操作步骤
    • 通过配置类向容器中注入其他的 IRule 接口实现类
      • 注意配置类位于 SpringBoot 应用能扫描到的路径

    配置类注入 其他的IRule 实现类

@Configuration
public class OrderConfig {
    @Bean
    @LoadBalanced
    public RestTemplate restTemplate(){
        RestTemplate restTemplate = new RestTemplate();
        List<HttpMessageConverter<?>> list = restTemplate.getMessageConverters();
        // 替换 StringHttpMessageConverter 的默认字符集
        for (HttpMessageConverter<?> httpMessageConverter : list) {
            if(httpMessageConverter instanceof StringHttpMessageConverter) {
                ((StringHttpMessageConverter) httpMessageConverter).setDefaultCharset(StandardCharsets.UTF_8);
                break;
            }
        }
        return restTemplate;
    }
    
   // 注入随机策略 RandomRule
    @Bean
    public IRule rule(){
        return new RandomRule();
    }
}

2.2.2 局部注入Ribbon的负载均衡策略（单独客户端使用）​

  • @RibbionClient

    • 作用: 在客户端访问指定微服务的时候优先加载 自定义的Ribbon配置类，从而使配置生效
      • 会体会默认的全局配置
      • 注意 : 自定义配置类 不能放在@ComponentScan 所扫描的包以及子包下 !!!( 比如 @SpringBootApplication中的扫描路径 )
    • @RibbonClient 的属性
      • name|value : 访问的微服务的服务名(serviceName)
      • configuration : 自定义的配置类
  • 操作步骤
    1. 创建配置类,配置类中注入IRule接口的实现类，并且将配置类放在 @ComponentScan 扫描不到的路径
    2. 在启动类上标注 @RibbonClient | @RibbonClients

①、包结构

②、自定义的配置类注入IRule实现类

@Configuration
public class MyRibbonRule {

    @Bean
    public IRule rule(){
        return new RandomRule();
    }
}

③、SpringBoot启动类上使用 @RibbonClients注解

@SpringBootApplication
@EnableDiscoveryClient
// @RibbonClients 可以同时为多个客户端服务注入规则策略
@RibbonClients(
       {
               @RibbonClient(name="cloud-payment-service-consul",configuration = MyRibbonRule.class)
       }
)
public class Order80 {
    public static void main(String[] args) {
        SpringApplication.run(Order80.class,args);
    }
}

2.3 Ribbon的“饥饿加载”​

  • 默认情况下,Ribbon 是"懒加载" 的 ,当通过Ribbon 第一次访问其他微服务的API时 , 才会创建一个RibbonClient
    • 通过 "饥饿加载" , 可以在服务启动的时候 , 就可以指定启动的时候创建哪些RibbonClient
  • 操作步骤 ?
    • 修改application.yml
      • ribbon.eager-load.enable
        • 开启饥饿加载
      • ribbon.eager-load.clients
        • 指定使用哪些微服务的时候 , 使用饥饿加载 
ribbon: 
  eager-load:
    enabled: true  # 开启饥饿加载
    clients: user-center  # 指定使用 user-center 微服务的时候,使用饥饿加载
